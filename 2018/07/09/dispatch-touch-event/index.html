Android触摸事件传递过程简单梳理

发表于 2018-07-09 00:12 | 更新于: 2018-07-09 00:12 | 分类于 Tech

最近蛮心烦的，所以就给自己找了点事儿，简单看了下Android触摸事件传递机制，然后简单做个笔记。

  1. ViewGroup#dispatchTouchEvent()

    触摸事件只要能传递到某个控件，那么这个控件的dispatchTouchEvent就一定会被调用

    首先如果是ACTION_DOWN事件就会重置一些上级触摸相关的状态记录

    接着检查是否需要拦截事件。

    如果是ACTION_DOWN事件或者mFirstTouchTarget不为空，并且不存在FLAG_DISALLOW_INTERCEPT这个flag，则会调用ViewGroup的onInterceptTouchEvent的方法来确定自己是否需要拦截事件，ViewGroup默认不拦截。

    接着如果不拦截并且不是取消事件则遍历子控件,跳过不能接收到事件的子控件，并且调用dispatchTransformedTouchEvent 将事件传递给子控件。

    如果这个方法返回了true，则说明事件被子View消耗了，那么newTouchTarget alreadyDispatchedToNewTouchTarget这两个变量会发生变化。

  2. ViewGroup#dispatchTransformedTouchEvent()

    这个方法是用来将事件传递给子控件的。

    • 当事件是ACTION_CANCEL
      • 如果子View为空的，就调用super的dispatchTouchEvent 即View的dispatchTouchEvent 。
      • 如果子View不为空，就调用子View的dispatchTouchEvent 来将事件传递给子View。
    • 不是ACTION_CANCEL 事件
      • 如果子View为空的，就调用super的dispatchTouchEvent 即View的dispatchTouchEvent 。
      • 如果子View不为空，就调用子View的dispatchTouchEvent 来将事件传递给子View。

    最终将dispatchTouchEvent 处理的结果返回回去。

  3. 继续step.1的ViewGroup#dispatchTouchEvent()

    这里分情况讨论:

    • 子View并消耗了事件，此时mFirstTouchTarget 这个变量的值不为空，那么就会走下面的else分支。因为Android系统是支持多点触控的，所以触摸到的View呢可能会有多个，所以这里就用了链式存储了，mFirstTouchTarget就是表头了(没具体看，错了不负责)，因此这里就会遍历这个触摸View并且将时间分发给它们。

    • 没找到合适的子View或者子View没有消耗事件，此时mFirstTouchTarget 这个变量就会为空，那里这里就会执行if这个分支。这个方法在上面已经解释了，当子View为空的时候就会调用super的dispatchTouchEvent ，简单讲就是把这个ViewGroup当做一个View来看待来响应这个触摸事件。

    到此为止，事件就分发下去了。

  4. View#dispatchTouchEvent()

    View的事件处理就简单得多，因为View作为最后一级存在，要么消耗，要么不消耗，不存在继续向下传递了。

    精简掉了一些与主题不相关的代码。

    ListenerInfo 这玩意儿呢保存的就是一些与View相关的监听器，包含OnClickListener OnLongClickListener OnTouchListener 等等相关的监听器。

    这里呢，依次判断这个保存相关监听器的对象是否为空，OnTouchListener是否为空，当前控件是否是Enable状态，如果前面都满足则会去调用OnTouchListener的onTouch方法，如果onTouch返回true。则表示View消耗了这个事件。如果为false则会继续去调用View的onTouchEvent方法。

    注意提前说一下，点击事件的onClick 方法是在onTouchEvent中调用的。

  5. View#onTouchEvent()

    这个里面，首先呢，即使控件状态是DISABLED 但是如果控件的Clickable或者LongClickable的，则还是会消耗这个事件。

    接着会判断控件的可点击状态，如果控件可点击，然后就会根据触摸事件的不同而执行不同的流程但是最终会返回true意味着事件被这个View消耗。如果控件不可点击则会直接返回false。

    如果控件是可点击的：

    • 那么在UP事件中会执行performClick 方法来执行onClick方法。
    • 在DOWN事件中会执行checkForLongClick 方法，在这个方法里面会创建一个触发onLongClick的Runnable对象并将其Post到消息队列中去，这个具体就不分析了，代码很简单。

    按照触发顺序来说，长按在前，点击在后。长按触发了后单击就不会触发了，因为长按触发了后mHasPerformedLongPress 这个变量的值就是为false了，这样的话就没法执行performClick 这个方法了。
