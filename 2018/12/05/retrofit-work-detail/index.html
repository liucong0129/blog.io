Retrofit执行流程.

发表于 2018-12-05 23:12 | 更新于: 2018-12-05 23:12 | 分类于 Tech

背景

因为项目需要自己实现一套缓存方案，并且在实现过程中出现了点小问题(实现之前并没有做可行性分析)，所以需要梳理下Retrofit的流程。

源码分析

Retrofit的执行过程主要分为4步:

  1. 构建Retrofit

  2. 通过Retrofit创建请求接口的实现类

  3. 通过调用实现类方法，方法返回表示HTTP请求的指定的对象实例。

    需要注意的是，这个返回的对象是在请求接口里面定义的。

    这个返回对象必须要有对应的CallAdapterFactory支持，不然Retrofit会抛异常。

  4. 激活返回的对象实例，开始发送请求。

    不同的对象激活方式不同，例如Retrofit内置的Call可以直接execute进行激活，而RxJavaCallAdapterFactory支持的Observable则需要调用subscribe进行订阅才能激活。

  5. 回调解析后的结果

下面的源码流程分析主要以Observable返回类型分析。

1.构建Retrofit

1.1.Retrofit.Builder.build

首先检查callFactory是否有设置，如果没有就使用OkHttpClient。

接着检查callbackExecutor，如果请求方法返回类型是Call，那么callback回调结果就会通过这个线程池来执行，这个线程池默认实现是啥并不重要，因为我们的请求方法返回值设定为Observable，此时是不会使用到这个线程池。

再接着在CallAdapterFactory集合里面增加一个默认的CallAdapterFactory，这个CallAdapterFactory是用来处理Call这种返回类型的。

最后根据这些数据创建Retrofit对象

2.通过Retrofit创建请求接口的实现类

2.1.Retrofit.create

通过Java内置的动态代理机制创建了一个请求方法接口的代理实现类。然后把这个代理类返回回去。

3.通过调用实现类方法，方法返回表示HTTP请求的指定的对象实例。

3.1.httpService.rxGetUserInfo

这个方法具体调用过程就需要看动态代理中的InvovcationHandler里面是怎么实现的了。

首先先描述下整个代码执行逻辑：

首先判断如果方法是Object里面的方法则直接调用。如果是当前平台的默认方法则将方法转交给平台自己去调用。如果都不是那就是我们自己定义的接口方法了。

然后去反射解析拿到这个方法相关的数据封装成一个ServiceMethod对象，然后将这个对象以及方法参数封装成OkHttpCall对象，接着调用ServiceMethod中的callAdapter的adapt将OkHttpCall适配成我们自定义的返回对象。

接着再分析loadServiceMethod这个方法看怎么解析那些请求方法以及需要解析哪些数据。

3.2.Retrofit.loadServiceMethod

众所周知，Java反射是很消耗性能的，所以理所当然的会对反射结果处理后的数据进行缓存。首先会从缓存serviceMethodCache中尝试获取，如果没有就会去解析并缓存，如果有就直接获取返回。具体的反射解析过程就是ServiceMethod.Builder.build方法了。

接下来就是分析ServiceMethod的构建过程了。

3.3.ServiceMethod.build

这个方法里面做了许多事情，但是执行逻辑却是很清晰的。

构建ServiceMethod大致可以分为4步：

1.创建CallAdaper；

2.创建ResponseConverter；

3.解析请求接口方法上面的注解；

4.解析请求方法的参数及其上面的注解；

上面步骤执行完成后，将这些数据构建ServiceMethod对象并返回。

3.3.1.创建CallAdaper
3.3.1.1.ServiceMethod.createCallAdapter

构建ServiceMethod的第一步：创建CallAdaper。

这个CallAdapter是用来将okhttp3.Call转换成我们自定义的返回类型。

首先拿到返回参数类型，然后拿到方法上面的注解，然后调用retrofit的callAdapter方法来创建一个合适的CallAdapter。

3.3.1.2.Retrofit.callAdapter

这个方法会遍历所有的callAdapterFactorys，看是否有能处理当前处理的接口方法返回值类型的CallAdapterFactory。具体这个过程会创建哪种CallAdapter则需要拿具体的CallAdapterFactory来分析。如果遍历所有的CallAdapterFactory发现还是办法处理这种返回值类型，就会抛出异常。

3.3.2.创建ResponseConverter

构建ServiceMethod的第二步：创建ResponseConverter。

这个玩意儿就是将服务器响应的ResponseBody类型转换成成我们想要的数据类型，也就是转换成在定义请求方法是返回类型的参数化类型这种类型。

这个里面直接调用retrofit的responseBodyConverter方法，并传入接口方法的注解以及返回类型Type。

这个方法会遍历所有的converterFactories，看是否有能处理当前处理的接口方法返回值的参数化类型的ConverterFactory。具体这个过程会创建哪种Converter则需要拿具体的ConverterFactory来分析。如果遍历所有的ConverterFactories发现还是办法处理，就会抛出异常。

3.3.3.解析请求接口方法上面的注解
3.3.3.1.ServiceMethod.parseMethodAnnotation

构建ServiceMethod的第三步：解析请求方法上面的注解。

这一步会遍历方法上面所有的注解，然后一一进行解析。

请求方法上面可能有许多的注解，并且解析的大多都是调用parseHttpMethodAndPath这个方法进行解析。

如果注解是Headers那么就会调用parseHeaders进行解析。

其他的就不一一赘述了。

这里我们就以GET注解作为例子进行分析。

3.3.3.2.ServiceMethod.parseHttpMethodAndPath

这个是用来解析Http的请求方法以及相对URL和相对URL里面包含的参数什么的。

3.3.4.解析请求方法的参数及其上面的注解
3.3.4.1.ServiceMethod.parseParameter

构建ServiceMethod的第四步：解析请求方法的参数上面的注解。

首先遍历方法参数的注解，然后去解析方法参数类型以及注解。

这里需要注意的是：

  • 方法参数类型好像不能包含TypeVariable和WidecardType这，两种参数化类型？？目前暂时还不清楚这两种Type到时是个啥。这个之后得找个时间研究一下。
  • 方法参数类型上面必须要添加注解并且只能是Retrofit的注解。Retrofit相关的注解例如Path、Query、Field等等。

然后开始去解析方法参数。

解析方法主要是解析参数的注解，而这些注解必须是Retrofit里面提供的，不能是别的，不然Retrofit会抛出异常。

能用到的注解有:Url Path Query QueryMap Header HeaderMap Field FieldMap Part PartMap Body

其中带Map的注解，参数类型必须为<? extends Map>，key必须为String类型,而value则根据注解的不同而不同，不过大多也是String类型。

这里就以Path作为例子进行分析。

3.3.4.2.ServiceMethod.parseParameterAnnotation

先拿到path注解里面的值，接着通过当前参数的参数类型以及注解通过去创建一个Converter，这个Converter就是将参数的Java对象转化成String。

接着去看那个Converter怎么获取。

还是惯例遍历converterFactories，调用stringConverter来创建Converter。如果遍历完毕没找着就是用内置的一个Converter实例，这个实例会直接调用参数的toString方法将对象转换成字符串。

举个栗子:

>

这个converter就是将uid转化成String类型，如果没有针对特定类型的Converter那么就会默认使用ToStringConverter直接调用这个uid对象的toString方法直接转换成字符串。

3.4.CallAdapter.adapt

拿到当前调用的方法对应的ServiceMethod对象实例后，接着将前面获取到的ServiceMethod对象实例和方法调用参数封装成一个OkHttpCall对象实例。

接着调用CallAdapter的adapt方法，将这个Call适配成我们请求方法里定义的返回类型，这个就是CallAdapter的功能所在。

在这里需要注意的是这个Call是retrofit2.Call，而不是okhttp.Call，两者的关键在这里可以简单说一下，retrofit2.Call和okhttp.Call两者很像是代理模式中的代理者和被代理者，其中retrofit2.Call像代理者角色，而okhttp.Call则像被代理者角色，在外部激活retrofit.Call时，retrofit.Call就会激活okhttp.Call。等okhttp.Call执行完毕后拿到okhttp.Response响应之后，retrofit.Call就会将这个响应解析成retrofit.Response。外界是不知道也不用关心retrofit.Call里面具体做了什么。

具体CallAdapter是如何adapt的，这个就依赖于具体的CallAdapter，现在是分析主要流程，这里就不展开了， 后面再统一拿具体实例分析一波。

4.激活返回的对象实例，开始发送请求

4.1.OkHttpCall.execute

不管CallAdapter将Retrofit.Call adapt成返回类型，最终激活这个返回类型的时候都会调用Retrofit.Call的execute方法来发送请求。

省略了部分代码。

这里会直接调用createRawCall方法创建一个okhttp3.Call对象实例。

最后执行这个call，执行结束后解析返回的结果。

这个OkHttpCall相当于是对okhttp3.Call进行了代理。

4.2.OkHttpCall.createRawCall

省略了部分代码。

这个里面通过serviceMethod的toRequest方法将方法参数转化成一个Request对象。具体怎么转化暂时先忽略，只需要知道调用这个方法后就创建好了Request对象。

接着调用serviceMethod的callFactory的newCall方法来创一个okhttp3.Call对象实例。

这里的callFactory如果我们没有设置，那么默认的CallFactory就是OkHttpClient了。这里一般不会自己再去实现各CallFactory，都是直接使用OkHttpClient。

4.3.OkHttpClient.newCall

直接new了一个RealCall对象然后返回。

到了这里，createNewCall方法就执行结束了。

4.4.okhttp3.Call.execute

上一步createRawCall返回的是OkHttpClient里面的Call实现类RealCall。接着会执行这个Call的execute发送请求。

到了这里就开始真正的请求网络了，

okhttp3.Call执行具体网络请求的过程会另外单独拿出来分析，这里只需要知道okhttp3.Call的execute方法执行完成后就拿到数据了。

4.5.OkHttpCall.parseResponse

网络请求完毕后OkHttpCall会对这个网络请求结果进行解析。

根据响应码执行不同的逻辑。

如果响应码小于200或者大于或等于300则表示此次请求失败，然后构建一个表示请求错误的retrofit2.Response并返回。

这里先部分展示retrofit2.Response里面包含的一些字段：

>

如果响应码为204或者205则直接构建一个请求成功但是body为空的retrofit2.Response并返回。

如果响应码都不满足前面的条件，那么最后就会调用serviceMethod的toResponse方法将okhttp3.Response中的ResponseBody转换成我们需要的数据类型。

4.6.ServiceMethod.toResponse

这个方法直接用Converter将okhttp3.Response中的ResponseBody转换成我们需要的数据类型。这个数据类型就是方法返回类型的参数化类型的实际类型。

例如我们返回值是Observable，那么这里的converter就是将ResponseBody转换成ResponseEntry。

怎么转换则需要看具体的Converter的convert的具体代码逻辑。这个后续再分析。

5.回调解析后的结果

转换成功后会将转换后的Java对象实例返回回去。

具体怎么回调回去这个就依赖于不同返回类型的具体实现了。

例如，常用的返回类型是Observable和retrofit2.Call，Observable是通过Subscriber回调，而retrofit2.Call则是通过retrofit2.Callback回调。
