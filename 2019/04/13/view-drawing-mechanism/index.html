View绘制流程源码分析

发表于 2019-04-13 23:47 | 更新于: 2019-06-11 17:12 | 分类于 Tech

上一篇分析了Activity从启动到进入前台整个流程，其中并没有分析Activity的从不可见到可见的过程，这一片就会继续上次分析过程来分析Android中的View的绘制流程。Activity在进入resume后会调用makeVisible方法使Activity可见。具体的测量和布局则通过LinearLayout来具体分析。

1.源码分析

1.1.Activity.makeVisible()

如果Activity中承载视图的Window还没有添加视图，就会调用ViewManager的addView来将根视图添加到Window中。

getWindowManager获取的ViewManager实际类型为WindowManagerImpl，接着会调用addView来将根视图添加到Window中。

1.2.ViewManager.addView(…)

需要注意的是这里的ViewManager实际类型为WindowManagerImpl。

接着会继续调用mGlobal的addView方法，这个mGlobal类型为WindowManagerGlobal。这个mGlobal主要是用来与WMS进行通信。

1.3.WindowManagerGlobal.addView(…)

首先创建一个ViewRootImpl对象实例root，最后通过root的setView方法将view与ViewRootImpl进行关联。

1.4.ViewRootImpl.setView(…)

首先在添加到WindowManager之前先进行layout，完成后会将此view的Window添加到WindowManager中。

添加成功后就调用view的assignParent方法将此ViewRootImpl为其父视图。

这篇只是关心View的绘制流程，因此这里将requestLayout执行完成后，后续将Window添加WindowManager以关联父视图过程则直接忽略。

1.5.ViewRootImpl.requestLayout()

这里先调用checkThread()方法来检查当前是否是UI线程。

接着调用scheduleTraversals方法来遍历视图。

1.6.ViewRootImpl.scheduleTraversals()

这里会调用Choreographer的postCallback方法，在合适的时机会执行mTraversalRunnable里面封装的操作。

1.7.ViewRootImpl.TraversalRunnable.run()

调用doTraversal方法开始遍历视图

1.8.ViewRootImpl.doTraversal()

接着会调用performTraversals开始执行遍历操作。

1.9.ViewRootImpl.performTraversals()

View的绘制流程主要有三步:测量、布局以及绘制。

其中对应的方法分别为performMeasure、performLayout、performDraw。

1.10.ViewRootImpl.performMeasure(…)

这个mView实际上是Activity中的根视图DecorView。

这里会调用其measure方法，来对mView进行测量。

1.11.View.measure(…)

final方法，不能覆盖。

这个measure方法规定好了测量流程(模板方法模式)。

因为不同的控件测量规则是不同的，所以onMeasure方法需要子类自行实现。

具体的测量过程则通过LinearLayout来举例进行分析。

1.12.LinearLayout.onMeasure(…)

LinearLayout可以设置两种方向:VERTICAL和HORIZONTAL。这里以垂直方向进行分析。

1.13.LinearLayout.measureVertical(…)

遍历所有子视图，然后调用measureChildBeforeLayout方法对子视图进行测量。当所有子视图测量完成后，LinearLayout会确定自己的宽高并调用setMeasuredDimension保存。

1.14.LinearLayout.measureChildBeforeLayout(…)

直接调用measureChildWithMargins方法测量子视图。

1.15.ViewGroup.measureChildWithMargins(…)

首先调用getChildMeasureSpec方法，根据父视图的MeasureSpec和子视图的LayoutParams来创建子视图的MeasureSpec。

最后子视图的宽高MeasureSpec创建完成后将其传递给子视图，子视图会根据这个MeasureSpec测量自己的宽高。如果LinearLayout中有容器类的子视图，那么容器类会先测量其子视图，然后等所有的子视图测量完成后再确定自己的宽高。

1.16.ViewGroup.getChildMeasureSpec(…)

这里会根据父视图的MeasureSpec和子视图自身的LayoutParams共同决定子视图的MeasureSpec。

简单描述下规则:

当父视图的测量模式为精确模式(EXACTLY)时，如果子视图的布局参数中宽或者高为特定尺寸，那么子视图的测量模式则也为精确模式，并且值为特定尺寸；如果子视图的布局参数中宽或者高为MATCH_PARENT，那么子视图的测量模式则也为精确模式，并且值为父视图的尺寸；如果子视图的布局参数中宽或者高为WRAP_CONTENT，那么子视图的测量模式将为AT_MOST，并且值为父视图的尺寸，这意味着子视图的宽高将根据自身内容调整，但是最大尺寸将不会超过父视图的尺寸。

当父视图的测量模式为AT_MOS模式时，如果子视图的布局参数中宽或者高为特定尺寸，那么子视图的测量模式将为精确模式(EXACTLY)，并且值为特定尺寸；如果子视图的布局参数中宽或者高为MATCH_PARENT，那么子视图的测量模式将为AT_MOST，并且值为父视图的尺寸，这意味着子视图的宽高将随着父视图的宽高而调整，但是不会超过父视图的尺寸；如果子视图的布局参数中宽或者高为WRAP_CONTENT，那么子视图的测量模式将也为AT_MOST，并且值为父视图的尺寸，这意味着子视图的宽高将根据自身内容调整，但是最大尺寸将不会超过父视图的尺寸。

当父视图的测量模式为UNSPECIFIED时，这意味着父视图不对子视图做任何限制，子视图想怎么浪就怎么浪。如果子视图的布局参数中宽或者高为特定尺寸，那么子视图的测量模式则也为精确模式，并且值为特定尺寸；如果子视图的布局参数中宽或者高为MATCH_PARENT，那么子视图的测量模式将也为UNSPECIFIED，这意味着子视图想要多大就多大；如果子视图的布局参数中宽或者高为WRAP_CONTENT时，情况和MATCH_PARENT完全相同。

最后将子视图的测量模式和测量值组合成一个MeasureSpec对象实例并返回。

整理成表格则如下所示：

子视图\父视图 EXACTLY AT_MOST UNSPECIFIED
SPECIFIC_SIZE EXACTLY/SPECIFIC_SIZE EXACTLY/SPECIFIC_SIZE EXACTLY/SPECIFIC_SIZE
MATCH_PARENT EXACTLY/PARENT_SIZE AT_MOST/PARENT_SIZE UNSPECIFIED/PARENT_SIZE
WRAP_CONTENT AT_MOST/PARENT_SIZE AT_MOST/PARENT_SIZE UNSPECIFIED/PARENT_SIZE

1.17.performLayout(…)

这里会直接调用根视图的layout方法进行布局。

1.18.View.layout(…)

onLayout方法是个抽象类，一般来讲V只有ViewGroup类的容器需要实现onLayout方法来布局子控件。

具体的onLayout则通过LinearLayout来举例进行分析

1.19.LinearLayout.onLayout(…)

同理，布局也分两种方向。这里还是以垂直方向布局进行说明。

1.20.LinearLayout.layoutVertical(…)

布局过程，简单讲就是计算子视图在在父视图中的位置(四顶点)，然后调用子视图的layout方法。

只需要确定顶点的位置，那么视图的位置就确定了。

这里根据layout_gravity计算出子视图的left，然后计算出top，子视图的right可以通过left和measuredWidth来确定，bottom同理。

需要注意的是，这四个顶点的坐标是以父容器的左上角为原点的相对坐标系(下为y正轴，右为x正轴)的，而不是以屏幕为原点的决定坐标系产生的。

四个顶点计算出来后就会调用setChildFrame方法对子视图进行布局。

1.21.LinearLayout.setChildFrame(…)

这个方法直接调用子视图的layout方法进行布局。

1.22.View.layout(…)

子视图会调用setFrame来保存位置信息。

1.23.View.setFrame(…)

保存好四个顶点信息。

需要注意的是，视图的宽是通过mRight和mLeft计算出来的。高同理。

1.24.ViewRootImpl.performDraw(…)

等测量和布局阶段完成后就会调用draw方法进行绘制。

1.25.ViewRootImpl.draw(…)

如果支持并启动了硬件绘制，就使用硬件绘制，否则使用软件绘制(CPU)。这里分析软件绘制。

1.26.ViewRootImpl.drawSoftware(…)

先通过Surface来获取一个Canvas对象实例，然后再调用根视图的draw方法开始绘制。

1.27.View.draw(…)

View中的draw方法中有注释描述了绘制流程:

1.Draw the background;
2.If necessary, save the canvas’ layers to prepare for fading;
3.Draw view’s content;
4.Draw children;
5.If necessary, draw the fading edges and restore layers;
6.Draw decorations (scrollbars for instance).

2.总结

主要是想通过了解View的绘制流程从而更好地理解自定义View。

View的绘制流程主要是分三步：measure、layout、draw。

测量阶段主要是确定视图的宽高，如果是容器类视图，那么会先确定子视图的宽高，然后根据子视图的宽高来确定自己的宽高。不同的视图一般来讲宽高的测量稍微有些区别，所以视图一般会重写相关方法来实现测量逻辑。

布局阶段主要是确定视图的位置，准确来讲是容器类视图来确定子视图的位置。一般来讲容器类视图会重写相关方法来实现自己的布局逻辑。例如LinearLayout，RelativeLut和GridLayout等等，都各自实现了其布局逻辑。

绘制阶段主要是绘制视图。将视图和一些装饰性的东西绘制到Canvas上面，最后通过硬件或者CPU进行绘制。
