RecyclerView源码简单分析2

发表于 2018-10-21 | 更新于: 2018-10-21 | 分类于 Tech

源码分析的场景为第一次填充整个屏幕(数据量足够的话)

源码版本为support-v7-26.1.0

首先将几个重点的方法按照调用流程列出来

  • RecyclerView.onMeasure()
  • LayoutManager.onMeasure()
  • RecyclerView.dispatchLayoutStep1()
  • RecyclerView.dispatchLayoutStep2()
    • LayoutManager.onLayoutChildren()
      • LinearLayoutManager.updateAnchorInfoForLayout()
      • LinearLayoutManager.fill()
        • LinearLayoutManager.layoutChunk()
          • LayoutState.next()
          • LinearLayoutManager.measureChildWithMargins()
          • LinearLayoutManager.layoutDecoratedWithMargins()
  • LayoutManager.setMeasuredDimensionFromChildren()

1.RecyclerView#onMeasure()

如果没有设置LayoutManager那么会调用默认的测量实现并结束测量流程。默认测量的代码很简单，这个不是重点就直接忽略了。

LinearLayoutManager默认开启了自动测量mLayout.mAutoMeasure=true，则会将测量流程委托给LayoutManager。LinearLayoutManager的测量还是使用了默认实现。接着会拿到RecyclerView的宽高测量模式，若都为精确模式MeasureSpec.EXACTLY则结束测量流程。

第一次加载的时候mState.mLayoutStep是State.STEP_START，因此会调用dispatchLayoutStep1方法。这个方法相对于我要分析的内容不够重要，暂时可以不用去管。查看源码的时候如果纠结每行代码的含义那么就非常困难了。

接着调用dispatchLayoutStep2开始布局第二个步骤。

这个步骤就会开始真正的布局。这一步可能会多次执行。

这里调用了LayoutManager的onLayoutChildren()方法将布局流程交给了LayoutManager。

2.LinearLayoutManager#onLayoutChildren

这个方法就是真正来测量子View的。RecyclerView将布局操作交给了LayoutManager，这样子View如何布局则取决于LayoutManager的这个布局策略。

布局算法简单描述：

  1. 寻找锚点坐标以及位置
  2. 从锚点位置开始向顶部开始填充
  3. 从锚点位置开始向底部填充

寻找锚点。

mAnchorInfo为布局锚点信息，包含了子控件在Y轴上起始绘制偏移量（coordinate），ItemView在Adapter中的索引位置（position）和布局方向（mLayoutFromEnd）——这里是指start、end方向。

计算锚点的坐标和位置比较复杂，主要逻辑在updateAnchorInfoForLayout方法中，暂时先忽略掉。

锚点找到了，下面就开始填充了。

填充过程大致为：先从锚点位置向顶部开始填充，接着再从锚点位置向底部开始填充。填充完毕如果还有剩余空间则继续填充。

下面接着看fill方法。

这里先简单描述下这个方法的执行流程

这个fill循环调用layoutChunk方法，每调用一次，根据剩余空间会减少，layoutState.mOffset的值也会跟着变化，这个值与子View的摆放位置有关。直到没有剩余空间或没有可布局的子View为止。

下面来具体分析layoutChunk方法。

先大致描述下这个方法的流程。

这个方法首先调用LayoutState的next(recycler)方法拿到View对象，接着将这个View添加到RecyclerView中。然后调用measureChildWithMargins对这个View进行测量，测量完毕后就计算这个View的left、top、right以及bottom，计算完毕后调用layoutDecoratedWithMargins来摆放这个子View。

下面看LayoutState的next方法看如何获取view的。

因为是第一次加载，所以mScrapList是为null的。因此，这个View就需要调用recycler的

getViewForPosition方法来获取了。

Recycler是RecyclerView的一个内部类，用于管理scrapped和detached itemView的复用。需要注意的是，缓存的单位为ViewHolder。scrapped view是指那些依旧关联着RecyclerView但是已经被标记为移除或重用的view。

大致描述下这个方法的执行流程：

  1. If there is a changed scrap, try to find from there；
  2. Find by position from scrap/hidden list/cache；
  3. Find from scrap/cache via stable ids, if exists；
  4. Find from ViewCacheExtension,if user has a implemention one;
  5. Find from RecyclerViewPool via type;
  6. if all of above cant give a ViewHolder ,just call some methods of Adapter to create ViewHolder。
  7. bind data if needed.
  8. perform initialization for view`s layoutParam。

前面3个是源码中有的，后面的是我自己写的。[英语渣]。

第6部中，最终会调用Adapter的onCreateViewHolder方法。

下面这段是引用网上别人的解析。

根据列表位置获取ItemView，先后从scrapped、cached、exCached、recycled集合中查找相应的ItemView，如果没有找到，就创建（Adapter.createViewHolder()），最后与数据集绑定。其中scrapped、cached和exCached集合定义在RecyclerView.Recycler中，分别表示将要在RecyclerView中删除的ItemView、一级缓存ItemView和二级缓存ItemView，cached集合的大小默认为２，exCached是需要我们通过RecyclerView.ViewCacheExtension自己实现的，默认没有；recycled集合其实是一个Map，定义在RecyclerView.RecycledViewPool中，将ItemView以ItemType分类保存了下来，这里算是RecyclerView设计上的亮点，通过RecyclerView.RecycledViewPool可以实现在不同的RecyclerView之间共享ItemView，只要为这些不同RecyclerView设置同一个RecyclerView.RecycledViewPool就可以了。

layoutChunk方法里面，当layoutState的next获取到了itemView之后，这个方法里面接着会调用measureChildWithMargins方法对返回的itemView进行测量。

这个测量方法需要注意mRecyclerView.getItemDecorInsetsForChild，这里就涉及到了RecyclerView的装饰部分了。

下面这段是引用网上别人的对ItemDecoration的描述。

RecyclerView内部就是在onDraw的时候执行ItemDecoration的onDraw，在draw的时候执行ItemDecoration的onDrawOver函数。在计算itemView的padding的时候将getItemOffsets得到的Rect加入其中，从而空出装饰内容的区域。

通过给RecyclerVIew添加RecyclerView.ItemDecoration，我们可以做出各种各样的装饰。这里暂时先了解一下，以后再分章节详细探索。

简单描述下这个方法的执行流程，首先拿到设置的ItemDecorations，然后统计计算出这些ItemDecorations总共在left、top、right、bottom这些方向上占用的空间，然后创建子View的MeasureSpec的时候将这些空间包含进去。最后如果子View需要进行测量就会调用子View的measure进行测量。

测量完成后会调用layoutDecoratedWithMargins进行拜访，在RecyclerView中。

RecyclerView将测量过程和布局过程都交由LayoutManager来处理了。

项目推送集成

发表于 2018-08-12 | 更新于: 2018-08-12 | 分类于 Tech

研究发现，大多都是针对不同设备集成不同推送平台，这样在最大限度里保持推送效果。

在知道信鸽可以很方便的集成厂商通道后，果断选择了信鸽。集成完毕后发现信鸽的推送延迟很高，例如单纯用信鸽的推送通道延迟甚至达到几十分钟，简直没谁了。听说个推也可以集成厂商通道，不过没有尝试，暂时先单独集成。等有空自己写个测试项目来测试看看效果。

最后集成方案：

  • 小米华为魅族等等，集成其自己的推送
  • 其他一律接入极光推送。

优点是厂商推送服务在对应的设备上是属于系统级服务，APP被杀死了依旧可以收到推送消息。

缺点也很明显，就是集成特麻烦，后台开发起来也特麻烦。

PS:公司APP用户中OPPO和VIVO的用户量特大，两者加起来甚至达到了40%。可惜这两家公司，一家暂不开放，一家尚在公测，现在就没法集成了。
